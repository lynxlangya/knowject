# Knowject 目标架构蓝图（2026-03-10）

本文档回答“Knowject 想做成什么”，以 `.agent/docs/inputs/知项Knowject-项目认知总结-v2.md` 为主要输入，并结合当前仓库现状做状态标注。它不是当前实现事实源。

## 1. 北极星

- 一句话定位：知项 · Knowject 是一个面向开发团队的项目级 AI 知识助手，把文档、代码与设计上下文沉淀成可持续复用的项目记忆。
- 目标用户：5 到 20 人的软件开发团队，重点覆盖开发、产品、设计的项目协作场景。
- 核心价值：
  - 降低项目知识分散带来的查询成本。
  - 降低知识传递对资深成员的依赖。
  - 让 AI 回答和协作建立在真实项目语境之上。
  - 把可复用资产沉淀为跨项目可复用能力。

## 2. 目标产品分层

### 2.1 全局层（目标态）

- 面向公司或团队级复用。
- 承载三类全局资产：
  - 全局知识库：跨项目共享的文档、代码、设计等知识资产。
  - 全局技能：可复用的能力单元，用于搜索、执行、分析、工作流触发等。
  - 全局智能体：预配置的 Agent 模板，可被多个项目引入。

### 2.2 项目层（目标态）

- 面向单个项目的上下文收敛和配置绑定。
- 承载：
  - 项目私有知识库。
  - 项目已引入的全局知识、技能、智能体。
  - 项目级 Prompt、成员关系、资源编排和协作规则。

### 2.3 对话层（目标态）

- 面向用户的实际协作入口。
- 通过“项目上下文 + 知识检索 + 技能调用 + 智能体推理”完成问答、分析和执行。
- 目标输出包括：
  - 多轮对话。
  - 来源引用。
  - 可见的工具调用过程。
  - 项目语境下的可执行建议。

## 3. 核心概念定义

### 3.1 Knowledge

- 定义：RAG 的数据来源，包含文档、代码、设计等项目上下文资产。
- 分层：
  - 全局知识库：跨项目共享。
  - 项目私有知识库：只属于单一项目。
- 目标职责：为对话和智能体提供可追溯检索依据。
- 当前状态：仓库里只实现了前端 Mock 目录与展示分层，未实现真实上传、解析、索引与检索。

### 3.2 Skill

- 定义：可被大模型按需调用的能力单元，不局限于 Function Calling。
- 目标形态：
  - Function Calling
  - API 调用
  - 深度 RAG 检索
  - 代码执行
  - 工作流触发
- 目标来源：
  - 用户自建
  - 团队全局共享
  - 公网引入
- 当前状态：仓库里只有“全局技能目录”和项目绑定 Mock，没有真实执行引擎。

### 3.3 Agent

- 定义：知识库 + 技能 + System Prompt + 角色设定的组合体。
- 目标职责：
  - 基于项目上下文决策何时检索知识。
  - 决策何时调用哪个 Skill。
  - 汇总结果并给出最终回答或动作建议。
- 当前状态：仓库里只有“全局智能体目录”和项目绑定 Mock，没有真实 Agent 调度链路。

### 3.4 三者关系

- 目标调用链：
  - 用户提问。
  - Agent 接收任务。
  - Agent 触发知识检索与技能调用。
  - LLM 在完整上下文下生成回答。
  - 前端展示文本、来源引用、工具调用过程。

## 4. 目标信息架构

- 全局层页面继续承载：
  - 首页 / 项目入口
  - 全局知识库
  - 全局技能
  - 全局智能体
  - 全局成员
  - 分析看板
  - 全局设置
- 项目层页面继续承载：
  - 项目概览
  - 项目对话
  - 项目资源
  - 项目成员
- 路由原则：
  - 与当前已落地的 canonical 路由体系保持一致，不再引入过期的 `/projects/:id` 命名。
  - 若在产品讨论中只讨论概念层，则不强绑定 URL。

## 5. 阶段能力边界

### 5.1 MVP 目标

- 账号与权限基础：
  - 注册 / 登录。
  - 项目创建。
  - 成员邀请与简单角色。
- 全局资产管理基础：
  - 全局知识库创建与文档上传。
  - 接入本地 Git 仓库。
  - 全局 Skill 基础管理。
  - 全局智能体创建与配置。
- 项目层基础：
  - 引入全局资产。
  - 项目私有知识库上传。
  - 默认项目助理 Agent。
- 对话基础：
  - 项目内多轮对话。
  - 流式返回。
  - 来源引用。
- 部署基础：
  - 可本地跑通的私有化部署方案。

### 5.2 第二阶段

- 模型可配置。
- GitHub / GitLab OAuth。
- Figma REST API 接入。
- Skill 公网引入。
- 更多 Skill 实现类型开放。
- 更完整的成员协作状态。
- 分析看板。

### 5.3 第三阶段

- MCP 协议支持。
- IM / Bot 集成。
- Electron 客户端。

### 5.4 明确不属于当前实现

- 以上所有能力目前都不应被表述为“已在仓库实现”。
- 当前仓库只完成了其中一部分产品壳层、信息架构和前端演示数据组织。

## 6. 目标技术方向

以下是目标方向，不是已经锁死或已落地的技术实现：

- 前端方向：
  - 延续当前 Web 前端壳层与项目信息架构。
  - 后续可能引入更正式的服务端状态管理和对话流处理，但不把具体库绑定为硬约束。
- 服务端方向：
  - 从演示 API 发展到项目、资产、对话、权限、检索、工具执行等正式后端模块。
- 数据层方向：
  - 需要结构化数据存储承载用户、项目、对话、权限与资产配置。
  - 需要向量检索基础设施承载文档和代码索引。
- AI 层方向：
  - 需要真正的检索链路、工具链路、Agent 编排与可解释输出。
- 部署方向：
  - 目标是私有化部署可落地，不依赖纯前端演示形态。

## 7. 当前仍待锁定的关键决策

- MVP 内置 Skill 先做哪几个，优先级如何排序。
- 用户自建 Skill 在 MVP 阶段开放到什么粒度。
- 项目创建时默认 Agent 的具体行为边界。
- 对话过程中如何选择 Agent，是入口级选择还是会话级切换。
- 全局智能体引入项目后，是否允许项目级覆写配置。
- 未来技术实现是否需要引入额外状态管理层，还是继续保持“能不用就不用”的最小复杂度原则。
