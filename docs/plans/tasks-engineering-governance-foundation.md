# 工程治理基线设计（长期治理机制）

状态：规划中（2026-03-20）  
优先级：P1  
阶段：Engineering Governance / Standards Foundation  
关联模块：`apps/platform` / `apps/api` / `apps/indexer-py` / `docker` / `scripts` / `docs` / `AGENTS.md`

当前结论：

- 本次任务定位是“建立长期治理机制”，不是发起一次全仓清债。
- 落地方式采用规范型治理，不把首批规则直接升级成 CI 硬门禁。
- 治理文档采用“一份总治理方案 + 五个专题标准包 + 一份评审清单”的分层结构。
- 第一阶段主线固定为“代码结构治理”；中文注释、配置安全、前端通用封装与文档同步作为配套标准同步建立。
- 现有 `docs/plans/*` 继续承担实施计划职责；后续由它们显式引用本治理体系，而不是被本体系替代。

---

## 一、目标

本计划的目标不是“让仓库立刻变干净”，而是让 Knowject 后续每一轮迭代都能按一致的工程标准判断：

- 什么算工程债；
- 什么问题必须在当前改动中处理；
- 什么问题可以延期，但必须留下例外记录；
- 代码、配置、前端封装与文档分别该同步到哪里。

本计划结束时，至少要达到以下结果：

- 仓库内有一套统一的工程治理总纲，不再依赖口头约定。
- 五条治理线具备统一模板：`目标 / 触发条件 / 坏味道判定 / 推荐动作 / 允许例外 / 文档同步要求`。
- 后续专项实施计划可以引用这套标准，而不是重新解释“为什么要做”。
- 日常评审可复用同一份治理检查清单，不再依赖临场补判断。

取舍依据：本次以 `Maintainability > Minimal Diff` 为主，但首批仅建立规范，不强行拉高自动化门禁。

## 三、治理模型设计

### 3.1 治理定位

本体系被定义为“工程治理层”，不是新的实施计划集合。它只负责三件事：

1. 定义什么叫问题。
2. 定义问题什么时候必须被处理。
3. 定义处理后要同步哪些文档与评审结论。

它不直接取代现有计划文档，不承载逐文件改造步骤，不直接绑定 CI 阻断策略。

### 3.2 规则分级

治理规则采用三层结构：

#### A. 红线规则

适用于安全、敏感配置、暴露面与事实文档同步错误这类不能被静默带过的问题，例如：

- secrets 明文入仓库；
- Docker / 数据库 / internal 接口暴露面失控；
- 架构级变更未同步事实文档或契约文档；
- 健康检查或错误响应泄露敏感信息。

这类规则即使暂未自动化，也必须在评审中被显式说明。

#### B. 核心治理规则

适用于本轮长期治理主线，覆盖：

- 巨石文件结构治理；
- 核心业务中文注释；
- 前端通用封装；
- 文档同步矩阵。

这类规则要求“必须有判断、必须有说明、必须有去向”，但可以基于例外机制延期处理。

#### C. 建议优化规则

适用于次级优化，例如局部命名、轻量重复、非关键样式收口等。  
这类规则进入 backlog，不作为当前阶段的评审主项。

### 3.3 例外机制

本体系允许例外，但例外必须可追踪。允许例外的场景仅建议保留三类：

- 当前改动是紧急修复，结构治理会明显放大回归风险；
- 相关模块已被现有专项计划承接，当前顺手治理收益低；
- 当前触点过小，不值得为了局部变更强行引入新抽象。

每次例外至少要记录：

- `位置`：涉及的文件或模块；
- `原因`：为什么本次不处理；
- `风险`：如果继续保留会造成什么问题；
- `去向`：预计回收到哪份计划或哪一阶段。

### 3.4 评审门禁

首批治理采用规范型门禁，不直接做强拦截。评审时至少回答以下问题：

- 当前文件是否已经越过巨石阈值并出现职责失真；
- 当前改动是否新增了可下沉为通用能力的前端逻辑；
- 核心业务流程是否缺少中文意图注释；
- 配置、环境变量或暴露面是否引入新的安全风险；
- 当前改动是否触发文档同步义务。

评审门禁先进入协作流程，再根据真实收益决定是否升级成自动化检查。

## 四、五个专题标准包

五个标准包统一使用相同模板：

- `目标`
- `触发条件`
- `坏味道判定`
- `推荐动作`
- `允许例外`
- `文档同步要求`

### 4.1 代码结构治理标准

这是第一阶段主线。

核心规则不是“文件超过 550 行必须拆”，而是“文件超过 550 行时，必须进入结构评估”。判断重点如下：

- 文件是否同时承担编排、状态、副作用、展示、adapter 多重职责；
- 页面层是否混入大量领域规则；
- service / repository façade 是否开始吞入具体实现细节；
- hook、组件、mapper、常量、helper 是否缺少边界分层。

推荐动作优先级：

1. 先拆职责边界；
2. 再拆共享 helper / adapter / hook / 展示组件；
3. 最后再观察文件行数是否自然回落。

第一批建议治理对象：

- `apps/platform/src/pages/project/ProjectChatPage.tsx`
- `apps/platform/src/app/layouts/components/AppSider.tsx`
- `apps/api/src/modules/knowledge/knowledge.repository.ts`

### 4.2 核心代码中文注释标准

目标不是“让所有代码都有注释”，而是给最容易被误读的核心逻辑补齐中文意图说明。

触发条件：

- 核心业务流程；
- 复杂条件分支；
- 状态机、补偿、回滚、幂等逻辑；
- 易误用 helper / façade；
- 涉及安全、兼容、迁移或边界约束的实现。

注释要求：

- 说明“为什么这样做”；
- 说明“边界在哪里”；
- 说明“哪些地方不能轻易改”。

禁止写法：

- 只复述代码表面行为；
- 每行都加废话式注释；
- 用注释掩盖可以通过重构消除的复杂度。

### 4.3 配置与安全治理标准

重点覆盖：

- Docker 配置；
- 数据库连接与暴露面；
- 环境变量与 secret 注入方式；
- internal 接口边界；
- 健康检查与错误响应暴露面；
- 开发态与生产态配置契约一致性。

优先消灭的风险：

- secrets 明文入仓库；
- 不必要的端口暴露；
- internal 能力边界不清；
- `.env*` 与 compose / 脚本语义漂移；
- 健康检查和日志泄露内部实现细节。

推荐动作以“配置审计清单 + 环境契约对照”为主，不在第一阶段追求安全自动化全覆盖。

### 4.4 前端通用封装治理标准

目标是明确“什么值得共享，什么不值得过早抽象”。

触发条件：

- 同一模式在两个以上页面稳定出现；
- 业务语义一致，而不是表面形态相似；
- 共享后能降低维护成本，而不是引入额外认知层。

优先治理对象：

- 通用状态 hook；
- 展示型共享组件；
- 领域 mapper / formatter；
- 主题 token 与高频样式基元；
- summary item、meta pill、表单资源选项等薄基元。

显式约束：

- 没有稳定复用证据时，不得引入新抽象层；
- 不允许为了“看起来更通用”牺牲当前业务可读性。

### 4.5 文档同步治理标准

目标是把“代码变化”稳定映射到“应该更新哪份文档”。

重点不是每次都更新所有文档，而是建立同步矩阵：

- 当前事实变化，更新 `current/`；
- 计划顺序变化，更新 `plans/`；
- 协议/边界变化，更新 `contracts/`；
- 协作入口或阅读顺序变化，更新 `README / handoff / AGENTS`。

本标准将复用并收敛现有文档同步边界：

- `AGENTS.md`
- `docs/README.md`
- `docs/current/architecture.md`

## 五、文档落点与目录衔接

### 5.1 推荐目录

后续正式落地治理标准时，建议新增：

```text
docs/standards/
  engineering-governance-overview.md
  code-structure-governance.md
  core-code-commenting.md
  config-security-governance.md
  frontend-shared-abstractions.md
  document-sync-governance.md
  review-checklist.md
```

本目录定位如下：

- `standards/`：长期规则；
- `plans/`：阶段性实施；
- `current/`：已落地事实。

### 5.2 为什么不继续放在 `plans/`

如果长期治理规则继续混在 `plans/`，半年后很容易被误认为阶段性旧文档。  
因此，本计划建议：

- 当前设计文档先落在 `plans/`，作为治理体系的设计依据；
- 真正实施时，再把总纲和五个标准包写入 `standards/`。

### 5.3 与现有计划的衔接方式

后续实施关系如下：

- `tasks-platform-frontend-refactor.md`
  - 作为“代码结构治理标准”的前端示范实施计划；
- `tasks-service-indexing-refactor.md`
  - 作为服务端 / 索引侧结构治理的既有实施计划；
- 后续若新增配置安全巡检、文档同步或通用封装专项，也继续放在 `plans/`，但必须引用对应标准包。

## 六、第一阶段输出物与里程碑

### 6.1 第一阶段输出物

第一阶段只交付以下结果：

- 一份总治理方案；
- 五份专题标准包；
- 一份评审检查清单；
- 对现有入口文档与实施计划的最小同步；
- 一份“第一批治理对象建议名单”。

第一阶段不要求：

- 立刻完成全仓清理；
- 立刻升级成 CI 门禁；
- 立刻替换所有历史计划。

### 6.2 里程碑建议

#### Milestone 0：治理设计冻结

- 完成本设计文档；
- 确认治理范围、规则分级、例外机制与目录落点。

#### Milestone 1：标准包与清单落地

- 创建 `standards/` 目录；
- 写入总纲、五个标准包与评审检查清单；
- 更新 `docs/README.md`、`.codex/README.md`、`AGENTS.md` 的入口说明。

#### Milestone 2：示范计划接线

- 让 `tasks-platform-frontend-refactor.md` 与 `tasks-service-indexing-refactor.md` 显式引用治理标准；
- 输出第一批治理对象建议名单与优先级说明。

#### Milestone 3：转入实施计划

- 使用 `writing-plans` 产出第一阶段实施计划；
- 固定验证与回滚方式，再进入代码与配置层执行。

## 七、验证方式

因为本轮采用规范型治理，所以验证不以自动化阻断为目标，而以“三类证据”作为完成标准：

### 7.1 文档证据

- 总治理方案、五个标准包、评审清单均已写入仓库；
- 入口文档已经能找到这些标准。

### 7.2 计划证据

- 现有实施计划已经开始引用治理标准；
- 后续专项计划不再重复写“为什么需要治理”。

### 7.3 评审证据

- 已形成可复用的治理检查清单；
- 在后续重构与安全巡检任务中可以直接套用。

## 八、风险与回滚

### 8.1 主要风险

- 规则写得过宽，导致执行层不知道什么才算必须处理；
- 标准包与实施计划边界不清，再次混写；
- 一开始就追求自动化门禁，反而让规范失去执行空间；
- 前端结构治理与配置安全治理互相抢优先级，导致计划发散。

### 8.2 止损策略

- 所有标准都必须写清 `触发条件` 与 `允许例外`；
- 先把规则收进文档和评审，而不是先做 CI 阻断；
- 第一阶段只固定“代码结构治理”为主线，其余四条作为配套标准并行建立。

### 8.3 回滚方案

若后续发现 `standards/` 目录引入成本过高，可临时把标准包保留在 `plans/` 作为过渡，但不推荐长期保留该形态。  
推荐回滚顺序：

1. 保留本设计文档；
2. 延后 `standards/` 目录创建；
3. 先让现有 `plans/*` 引用治理规则草案；
4. 待规则稳定后再迁出。

## 九、下一步

在本设计获得书面确认后，下一步不是直接改代码，而是：

1. 先创建 `standards/` 总纲、五个标准包与评审清单；
2. 再用 `writing-plans` 写第一阶段实施计划；
3. 最后再进入代码结构治理与配置巡检的执行阶段。
